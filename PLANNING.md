# NeuroLex

> AI-powered language acquisition system that lives inside Obsidian — 10 specialized modules, spaced repetition, grammar prerequisite tracking, avoidance detection, and adaptive session orchestration.

**Created:** 2026-04-02
**Type:** Application
**Stack:** Obsidian Plugin API + TypeScript, Claude Agent SDK, esbuild, MongoDB, Voyage AI
**Skill Loadout:** PAUL (managed build), /paul:audit (milestone reviews)
**Quality Gates:** test coverage, eval framework pass rates, behavioral compliance, prerequisite graph enforcement

---

## Problem Statement

NeuroLex solves the gap between shallow chatbot language tutors and principled language acquisition. Existing AI tutors offer undifferentiated conversation practice — they don't separate correction from instruction from planning from diagnosis. NeuroLex is architecturally opinionated: each of its 10 modules has a single responsibility, behavioral rules, and a defined interface.

**Who it's for:** One learner first (German, B1→B2), architected to serve many.

**Why build vs buy:** No existing tool combines spaced repetition, grammar prerequisite graphs, avoidance detection, and adaptive session orchestration in a single system — especially not one where the learner owns their data as Obsidian notes.

**Fork basis:** Claudian (`github.com/YishenTu/claudian`) — preserves its Claude Agent SDK integration, streaming, tool rendering, MCP infrastructure, and chat UI. Strips generic features, rebrands, and rebuilds as a dedicated language tutor.

## Data Model

Three-tier data architecture — each tier serves a different need.

### Tier 1: MongoDB — Engine Data

| Collection | Key Fields | Queried By |
|------------|-----------|------------|
| `learner_profile` | level per structure, L1 background, preferences, weekly snapshots | Diagnostiker, Architekt, Mentor |
| `srs_cards` | word, context sentence, semantic field, interval, ease factor, next review, history | Wortmeister |
| `grammar_progress` | per-structure: zone, mastery %, free-production accuracy, prerequisite status | Grammatiktrainer, Architekt |
| `error_log` | structure, type, tier, context, correction, timestamp, fossilization flag | Korrektor, Diagnostiker |
| `avoidance_tracking` | per-structure: opportunities, uses, ratio, status, workaround strategies | Diagnostiker |
| `session_history` | date, duration, modules used, focus structures, plan vs actual, engagement | Architekt, Mentor |
| `lernauftrag` | task, audience, register, deadline, required structures, readiness score | Architekt, Übungsmeister |

### Tier 2: Voyage Embeddings (via MCP Server)

- Error pattern clustering (Korrektor)
- Vocabulary coverage analysis (Kurator)
- Similar exercise retrieval (Grammatiktrainer)
- Contextual word suggestions (Wortmeister)
- Progress narrative comparison (Mentor)

| Writing samples + corrections | Schreibtrainer + Korrektor |
| Lernauftrag logs | Übungsmeister |

## API Surface

NeuroLex is an Obsidian plugin — no REST API. The "API surface" is MCP tool interfaces and module agent routing.

### Module Agent Interface

Each module is a custom agent (`.md` file) with:
- System prompt with module-specific behavioral rules
- Tool restrictions (which MCP tools this module can access)
- Model override (Opus for nuance-heavy modules, Sonnet for simpler tasks)

### MCP Server Tools

| MCP Server | Tools Exposed | Auth |
|------------|---------------|------|
| `neurolex-mongodb` | `query_profile`, `update_srs`, `log_error`, `get_avoidance`, `save_session` | Local connection string |
| `neurolex-voyage` | `embed_text`, `find_similar`, `compute_coverage`, `cluster_errors` | API key in env |
| `neurolex-tts` (v1.1) | `speak_text`, `set_voice`, `set_speed` | API key in env |

### Module Routing (Übungsmeister)

```
Learner ↔ Übungsmeister (main agent / conductor)
              ├── → Sprechtrainer (speaking + voice loop)
              ├── → Schreibtrainer (writing tasks)
              ├── → Wortmeister (vocabulary SRS)
              ├── → Grammatiktrainer (grammar exercises)
              ├── ← Korrektor (observes all output, corrects)
              ├── ← Kurator (ingests external texts)
              └── ↕ Intelligence Layer
                    ├── Diagnostiker (data + analytics)
                    ├── Architekt (session planning)
                    └── Mentor (summaries + dashboard)
```

## Security Considerations

- **Auth model:** None required — local-first plugin, single user, local MongoDB. No multi-user auth.
- **Secrets management:** MongoDB connection string and Voyage/TTS API keys stored in Obsidian plugin settings (encrypted at rest by Obsidian) and/or environment variables. Never hardcoded.
- **Input validation:** All learner input sanitized before MongoDB queries — parameterized queries only, no string concatenation.
- **Data sensitivity:** Learner error logs, avoidance patterns, and session history are personal data. Stays local. No telemetry, no cloud sync in v1.0.
- **MCP server isolation:** Each MCP server exposes only its defined tools. No shell access, no filesystem writes outside designated note folders.
- **OWASP relevance:** Injection (MongoDB query injection) is the primary concern. XSS not applicable (no web UI). CSRF not applicable (local plugin).

## Design Decisions

1. **Fork Claudian rather than build from scratch:** Claudian already has Claude Agent SDK integration, streaming, MCP infrastructure, and chat UI — 70% of the infrastructure we need. Rebuilding would take months.
2. **10 separate agents, not one monolithic prompt:** Single responsibility per module. Korrektor doesn't plan sessions, Architekt doesn't correct errors. This allows per-module model selection (Opus for nuance, Sonnet for simple tasks) and isolated testing.
3. **MongoDB + Voyage + Obsidian notes (three-tier data):** MongoDB for fast engine queries (SRS scheduling, error logs), Voyage for semantic intelligence (clustering, similarity), Obsidian notes for learner ownership (readable, searchable, annotatable).
4. **MCP servers for external services:** MongoDB and Voyage deployed as standalone MCP servers, not baked into the plugin. Keeps the plugin lightweight and services swappable.
5. **Grammar prerequisite graph with strict enforcement:** Never teach B4 before B1. The graph is the pedagogical backbone — modules query it, never bypass it.
6. **Avoidance detection over error tracking alone:** Errors are visible; avoidance is invisible and more dangerous. Usage ratio tracking with minimum data thresholds prevents false positives.
7. **German-only for v1.0:** Multi-language support deferred. Building for one language deeply is better than many languages shallowly.
8. **CLI-based inference (no API billing):** All inference runs through Claude Code or Codex CLI under the user's existing subscription. No per-token costs.
9. **Per-module model overrides:** Korrektor, Sprechtrainer, and Mentor use Opus (nuance-heavy). Wortmeister SRS and Diagnostiker queries use Sonnet (simpler tasks).
10. **One-Correction Rule as universal policy:** Overcorrection shuts learners down. One error per turn, prioritized by tier (communication-breaking → pattern-forming → stylistic).
